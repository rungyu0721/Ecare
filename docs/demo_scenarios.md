# E-CARE Demo Scenarios

這份文件整理偏鄉、山區、國家公園與常見高風險事件下，最適合展示 E-CARE 的 demo 對話。Demo 目標不是取代 119/110，而是展示系統如何在慌亂對話中快速整理救援需要的資訊，並判斷 119、110 或同步通報。

## Demo 1：國家公園步道迷路

使用者輸入：

```text
我們在國家公園步道迷路了，手機快沒電也沒訊號。
```

系統應判斷：

- 類別：山域水域救援
- 風險：High
- 建議：優先撥打 119
- 重點資訊：GPS 座標、步道名稱、最近地標、同行人數、手機電量、訊號狀況、天候與天色

期望回覆方向：

```text
這屬於需要 119 協助的山域救援情境。請先停在安全位置、保留手機電力；你現在有 GPS 座標、步道名稱或附近地標嗎？
```

展示重點：

- 系統不會只問一般地址，而會追問 GPS / 步道 / 地標。
- 手機快沒電會提高風險，並提醒保留電力。

## Demo 2：山上摔傷不能走

使用者輸入：

```text
朋友在山上摔落，腳疑似骨折，現在不能走。
```

系統應判斷：

- 類別：山域水域救援
- 風險：High
- `people_injured`: true
- `danger_active`: true
- 建議：撥打 119，提供座標、同行人數與傷勢

期望回覆方向：

```text
請立刻撥打 119，告知 GPS 座標或步道地標；先不要搬動疑似骨折或墜落傷者。你們同行幾個人，目前位置能安全停留嗎？
```

展示重點：

- 系統避免建議使用者自行搬動傷者。
- 會把「不能走」視為仍需救援的持續狀態。

## Demo 3：溪水暴漲受困

使用者輸入：

```text
我們溯溪時溪水暴漲，被困在溪谷旁邊。
```

系統應判斷：

- 類別：山域水域救援
- 風險：High
- `danger_active`: true
- 建議：撥打 119，不要冒險涉水或下切溪谷

期望回覆方向：

```text
請立刻撥打 119，先待在相對安全的位置，不要涉水或下切溪谷。你們有 GPS 座標、溪谷名稱、步道名稱或明顯地標嗎？
```

展示重點：

- 系統辨識「水域救援」不是普通醫療諮詢。
- 回覆會先阻止危險行動，再補位置資訊。

## Demo 4：失溫與手機低電量

使用者輸入：

```text
我們在山上有人失溫，手機快沒電了。
```

系統應判斷：

- 類別：山域水域救援
- 風險：High
- `people_injured`: true
- `danger_active`: true
- 建議：撥打 119，提供座標、同行人數、失溫狀況與手機電量

期望回覆方向：

```text
這是高風險狀況，請立刻撥打 119，開擴音並保留手機電力。請準備回報 GPS 座標、步道地標、同行人數，以及失溫者是否清醒、是否能正常呼吸。
```

展示重點：

- 系統會把失溫視為山域高風險。
- 不會假設附近有 AED。

## Demo 5：偏鄉家人急症

使用者輸入：

```text
我阿公在偏鄉家裡胸口很痛，冒冷汗，救護車可能不知道怎麼進來。
```

系統應判斷：

- 類別：醫療急症
- 風險：High
- `reporter_role`: 照顧者/家屬
- 建議：撥打 119，提供地址、明顯地標、路況與患者症狀

期望回覆方向：

```text
胸痛合併冒冷汗需要優先處理，請立刻撥打 119。請準備告知住家地址、附近地標、可通行的道路，以及阿公目前是否清醒、能否正常呼吸。
```

展示重點：

- 偏鄉不是只有山難，也包含救護車難以定位或抵達的急症。
- 系統會同時整理醫療症狀與路線/地標資訊。

## Demo 6：山區人身威脅

使用者輸入：

```text
我們在山屋附近有人拿刀威脅我們，旁邊有人跌倒受傷。
```

系統應判斷：

- 類別：暴力事件
- 風險：High
- `weapon`: true
- `people_injured`: true
- `danger_active`: true
- 建議：優先 110，因有人受傷同步 119

期望回覆方向：

```text
這是高風險人身威脅，請先離開對方視線並不要正面接觸。請立即撥打 110；因現場有人受傷，也需要同步撥打 119，並提供山屋名稱、GPS 座標或最近地標。
```

展示重點：

- 山區事件不一定都只打 119，系統會依照暴力/犯罪訊號判斷 110。
- 若同時有人受傷或受困，系統會提醒同步 119。

## Demo 7：地震後建築物倒塌

使用者輸入：

```text
地震後大樓倒塌，有人被壓住，旁邊還有瓦斯味。
```

系統應判斷：

- 類別：天然災害
- 風險：High
- `people_injured`: true
- `danger_active`: true
- 建議：優先 119，必要時同步地方災害應變單位；遠離倒塌與瓦斯危險區

期望回覆方向：

```text
這是高風險災害救援情境，請先遠離倒塌建物與瓦斯味區域，立刻撥打 119。請準備告知位置、倒塌範圍、是否有人被壓住，以及你現在是否在安全處。
```

展示重點：

- 系統會將地震、倒塌、被壓住判為天然災害 High。
- 回覆先要求遠離二次危險，再整理受困/受傷資訊。

## Demo 8：電梯受困

使用者輸入：

```text
我們困在電梯裡，門打不開，有人開始不舒服。
```

系統應判斷：

- 類別：受困救援
- 風險：High
- `people_injured`: true
- `danger_active`: true
- 建議：撥打 119 或請管理員同步通知消防，提供地址、樓層、電梯編號

期望回覆方向：

```text
請不要強行開門或攀爬，先保持通話與通風。請撥打 119 或通知管理員同步聯絡消防，告知地址、樓層、電梯編號，以及不舒服的人目前狀況。
```

展示重點：

- 電梯受困不會被誤判成天然災害。
- 系統會追問樓層、電梯位置、受困人數與是否有人不適。

## Demo 9：市區小孩走失

使用者輸入：

```text
我小孩在市區公園附近走失，現在找不到人，電話也聯絡不上。
```

系統應判斷：

- 類別：失蹤走失
- 風險：High
- `danger_active`: true
- 建議：優先 110；若有受困、受傷或進入山區水域，再同步 119

期望回覆方向：

```text
這是高風險走失通報，建議盡快撥打 110。請先整理最後看到的時間、地點、穿著、照片與聯絡方式；如果附近有水域、山區或可能受困受傷，也要同步 119。
```

展示重點：

- 市區走失會走 110，不會被山域救援規則吃掉。
- 若描述是國家公園或山區失聯，系統會改判山域水域救援。

## Demo 10：自殺危機 / 跳樓通報

使用者輸入：

```text
我朋友站在頂樓說要跳樓，現在只有我在旁邊。
```

系統應判斷：

- 類別：自殺危機
- 風險：High
- `danger_active`: true
- 建議：同步 119 與 110；保持陪伴與安全距離，不刺激、不拉扯

期望回覆方向：

```text
這是高風險自殺危機，請先保持安全距離陪著他，不要刺激或拉扯。請立刻同步撥打 119 和 110，告知位置、樓層、他是否在邊緣，以及現場是否有危險物。
```

展示重點：

- 系統會辨識「跳樓 / 自殺 / 輕生」為自殺危機 High。
- 即使有人陪伴，仍保持 High；若已離開危險位置，`danger_active` 可降為 false，但仍建議專業協助。

## Demo 講解重點

- E-CARE 不宣稱已派遣救援，只協助整理資訊並引導使用者聯絡 119/110。
- 系統會自行判斷 119、110 或兩者都需要：醫療/消防/天然災害/受困救援/山域水域救援偏 119，自殺危機同步 119 與 110，失蹤走失/暴力/犯罪/人身威脅偏 110，混合情境同步提醒。
- RAG / taxonomy 會把山域救援常見關鍵字注入對話，例如步道、溪谷、失溫、手機快沒電。
- 山區或偏鄉不假設附近有 AED；只有使用者明確說現場有 AED 時才進 AED 操作流程。
